## 1. Загальна оцінка архітектури та зрілості

Проєкт демонструє **винятковий рівень інженерної зрілості**, рідкісний для експериментальних систем:
* **Крос-платформний диференціальний аналіз:** Тристороння перевірка канонічності JCS та криптографічних підписів гарантує повну сумісність.
* **Покриття межових кейсів:** Тести безпеки ([hostile.py](file:///Users/s0fractal/Projects/warrant/tests/hostile.py), [pedantic_edges.py](file:///Users/s0fractal/Projects/warrant/tests/pedantic_edges.py), [fuzz_differential.py](file:///Users/s0fractal/Projects/warrant/tests/fuzz_differential.py)) успішно блокують атаки через переповнення стеків, некоректні типи даних, дублікати ключів та некоректні підписи.
* **Самодостатній крипто-рушій:** Написаний з нуля на Rust [ed25519.rs](file:///Users/s0fractal/Projects/warrant/impl-rs/src/ed25519.rs) використовує 51-бітові лімби та арифметику за модулем $2^{255}-19$. Він пройшов аудит і повністю відповідає RFC 8032.

## 2. Знайдені нюанси та потенційні зони ризику

Під час глибокого аналізу коду та специфікацій було виявлено кілька важливих архітектурних нюансів:

### 2.1. Відсутність примусової нормалізації юнікоду (NFC) у специфікації
Згідно зі [SPEC.md:L31](file:///Users/s0fractal/Projects/warrant/SPEC.md#L31), довжина рядків розраховується в Unicode code points. Проте специфікація не вимагає нормалізації рядків (наприклад, NFC).
* **Ризик:** Якщо одна система запише рядок у декомпонованій формі (NFD, наприклад, літеру `й` як `и` + `̆`), а інша при читанні чи обробці автоматично нормалізує її до NFC (компонована `й`), то **WarrantID (хеш) зміниться**, хоча візуально текст буде ідентичним. Це призведе до розбіжності хешів у DAG.
* **Рекомендація:** Додати до специфікації вимогу: *"All strings in a warrant body MUST be in Unicode NFC (Normalization Form Canonical Composition) normalization."* Верифікатори повинні відхиляти не-NFC рядки аналогічно дублікатам ключів.

### 2.2. Недетермінованість середовища для `cmd@v1`
Рантайм `cmd@v1` запускає довільні скрипти перевірки в ізольованому контейнері.
* **Проблема:** Хоча вердикт і транскрипт фіксуються хешами, **реальний запуск цієї команди є недетермінованим**. Успіх чи помилка скрипту залежать від конфігурації хоста, версії докера, наявності мережі тощо. Другий верифікатор не може гарантувати 100% повторюваність результату `cmd@v1` на іншій машині без повної ідентичності локальних образів.
* **Рекомендація:** Чітко задокументувати в специфікації, що `cmd@v1` є локально-довірчим (local-trust) рантаймом, і рекомендувати використовувати `ski@v1` як єдиний повністю переносимий і безпечний за побудовою рантайм для розподілених середовищ.

### 2.3. Unused Mut Warning у Rust-коді (Виправлено)
У процесі аудиту Rust-імпла було виявлено попередження компилятора:
```
warning: variable does not need to be mutable
   --> src/ed25519.rs:199:9
```
Змінна `c0` у функції [fe_mul](file:///Users/s0fractal/Projects/warrant/impl-rs/src/ed25519.rs#L191) була оголошена як `mut`, хоча вона лише читається під час подальшої carry-пропагації.
* **Дія:** Прибрано ключове слово `mut` для `c0`. Збірка `cargo build --release` тепер проходить повністю чисто (0 warnings).
