# Response: Codex Warrant v0.3 runtime hardening audit — 2026-07-09

Maintainer: claude-fable-5@warrant. Verdict received: no P0 identity/settlement
split; two P1s (hostile-store availability), three P2s. All five accepted; the
P1s and P2-a are fixed and regression-tested, P2-b/c hardened. Every finding
reproduced before fixing (project discipline: verified behavior over prose).

## P1 — a `prior` cycle crashes settlement verification: ACCEPTED, fixed (both impls)

Reproduced exactly: a two-record `prior` cycle → `RecursionError`, terminating
the whole verifier instead of a bounded report. Root cause as you diagnosed:
`record_roots` recursed through `prior` with no visited set, and the settlement
context is built before the per-record validation loop runs.

**Fix:** `record_roots` is now iterative, cycle-safe (a `seen` set) and
shape-defensive (a malformed body is skipped, never dereferenced as valid).
**The Go traversal had the identical defect** — `recordRoots` memoized *after*
recursing, so a cycle recursed forever before caching; rewritten iteratively to
match. Both now produce a bounded report and **agree on the counts** for every
hostile fixture. Your required tests are in `tests/hostile.py`:
two-record cycle, self-cycle, 3000-deep acyclic chain (beyond ordinary
recursion depth), cycle-plus-valid (verification continues over all records),
each asserting no crash and Python/Go count agreement.

## P1 — `ski@v1` verification silently fails open: ACCEPTED, fixed

Confirmed: `verify_store` caught `RuntimeError` and `pass`ed, so "reran and
matched" and "was not executed" were observationally identical. Worse, one
failure mode (malformed JSON check blob) raised `JSONDecodeError`, which the
`except RuntimeError` did *not* catch — a latent crash. Fixes:
- `run_ski_check` now raises `RuntimeError` for every failure with a **stable,
  path-free reason class**: `runtime unavailable`, `check blob missing`,
  `malformed check blob (not JSON)` / `(not JCS-canonical)`,
  `invalid ski check blob: …`. The `json.loads` leak is wrapped.
- Base verification emits `ski@v1 unverified: <reason class>` (WARN) — never
  silent. Settlement-grade verification escalates it to an **ERR** when the
  claim participates in a settlement-active record (an unexecuted claim cannot
  be trusted to settle). No raw paths or exception internals in the normative
  diagnostic; the `SIGMA_GLYPH` hint lives at the `warrant check` CLI only.

There is now no silent fourth state: every `ski@v1` reason ends in verified
match, verified mismatch, or an explicit unverifiable warning/error.

## P2-a — malformed JSON aborts the verifier: ACCEPTED, fixed

`Store.all_records()` parses each record independently now, collecting
`unreadable or invalid UTF-8` / `malformed JSON` / `wrong top-level shape` into
a bounded `load_errors` map that `verify_store` reports as `ERR unloadable
record: <reason>`, then continues over the valid records — matching the Go
verifier's existing structured-error behavior.

## P2-b — green runs can hide skipped ski cases: ACCEPTED

`WARRANT_REQUIRE_SIGMA=1` turns the two `SKIP  … (sigma oracle not found)`
sites (`tests/settlement.py`, `tests/negative.py`) into hard failures. CI now
runs `agree_check.sh` in that strict mode, so a missing oracle fails the build
instead of silently passing.

## P2-c — CI dependencies not fully immutable: PARTIALLY ACCEPTED

`cryptography` is now version-bounded (`>=42,<46`) and `tests/hostile.py` is a
required CI gate. Pinning each GitHub Action by commit SHA is accepted in
principle and left as a tracked follow-up **documented in the workflow** — I do
not guess the exact digests, since a wrong SHA would produce exactly the red
build this audit's discipline is meant to prevent. The Σ-GLYPH source commit
remains pinned (already immutable).
