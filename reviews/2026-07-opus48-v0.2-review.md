**Reviewer:** Claude (Opus 4.8, 1M)
**Role:** Adversarial reviewer — "run first, read second"; every headline finding reproduced locally.
**Timestamp:** 2026-07-07
**Scope:** `SPEC.md`, `impl/warrant.py`, `impl-go/main.go`, `tests/agree_check.sh`, `.github/workflows/ci.yml`, `examples/`.

---

## Verdict

**Solid prototype; the differential-correctness bar it sets for itself is unmet.** The content-addressing core is sound: WarrantID covers the whole canonical body, recomputation is enforced on read, tamper is detected, the actor-signature requirement holds, and the embedded Go Σ-GLYPH reducer reproduces the pinned ski@v1 vectors. The five SPEC §8 vectors pass byte-exactly in Python.

**But the project's own design rule (SPEC line 5) — _"two independent implementations MUST agree on every WarrantID and every verification outcome"_ — is violated in two concrete, reproduced cases**, and neither is caught by any test because `agree_check.sh` never actually compares the two implementations on valid inputs. Fixing the canonicalization divergence (F1) and adding a real differential harness (F4) would move this from "correct on the happy path" to "meets its stated bar."

---

## Findings (most severe first)

### F1 — MAJOR (borderline blocker): Python and Go compute different WarrantIDs for strings containing U+0008 / U+000C

**`impl-go/main.go:174-182` (`quoteJSONString`) vs `impl/warrant.py:39-41` (`canon`).**

Go serialises strings via `encoding/json` with `SetEscapeHTML(false)`. Go's stdlib encoder emits short escapes **only** for `\` `"` `\n` `\r` `\t`; every other control byte < 0x20 — including backspace (U+0008) and form-feed (U+000C) — becomes a six-char `\u00xx` escape. Python's `json.dumps(…, ensure_ascii=False)` emits the two-char `\b` and `\f` short forms, which is also what **RFC 8785 (JCS)** — the canonicalization SPEC §4 explicitly cites — mandates.

Reproduced (U+0008 = backspace):
```
Python canon("x" + U+0008 + "z")  ->  bytes:  " x \b z "        (JCS short escape, 2 chars)
Go   quoteJSONString(same)         ->  bytes:  " x \u0008 z "  (stdlib, 6 chars) — DIVERGES
```
So the two implementations produce different canonical bytes → different SHA-256 → **different WarrantID** for the same body. `subject.note`, `prose.text`, and `actor.id` are free-form strings, so these bytes are legal input. Each implementation "verifies OK" against its **own** recomputed ID, so this is a **silent** split of the content-addressing primitive — the worst failure mode for a hash-addressed format — and Go is non-conformant to the JCS it claims.

**Fix:** do not delegate canonical string output to `encoding/json`. Write an explicit JCS escaper in Go: short forms for `\b \t \n \f \r \" \\`, `\u00xx` (lowercase) for every other byte < 0x20. Add a differential fuzz over every byte 0x00–0x1F (see F4).

### F2 — MAJOR: `subject.note` length — Go counts bytes, Python counts code points → validators disagree on validity

**`impl-go/main.go:264` (`len(s) > 200`) vs `impl/warrant.py:78` (`len(note) <= 200`).**

SPEC §2 says note is "≤200 **chars**". A 150-character note of 2-byte code points is 300 bytes: Python accepts (150 ≤ 200), Go rejects (300 > 200). Directly violates "agree on every verification outcome," and Go is wrong per spec.

**Fix:** in Go use `utf8.RuneCountInString(s) > 200`.

### F3 — MAJOR (security): the key↔actor binding MUST-report is not implemented; forged-actor warrants verify clean

**`impl/warrant.py` verify/`why`; `impl-go/main.go` verify.**

SPEC §5: _"implementations MUST verify signatures against the stated key **and report the binding as unverified if no keyring is configured**."_ Neither implementation has a keyring concept, and neither ever emits an "unverified binding" marker. Verification only checks that some signature by the envelope-supplied `key` is valid and that its `actor` equals `body.actor.id`. Since the filer chooses the key, **anyone can produce a fully-valid warrant claiming to be any actor** (e.g. `agent-a@vendor1`), and both `verify` and `why` present it with zero caveat. For a format whose entire value is trustworthy decision records, presenting attacker-keyed records as authoritative without a warning is a real gap between the spec's MUST and the code.

**Fix (minimal):** emit a `WARN` such as `binding unverified (no keyring)` for every signature whose key is not confirmed against a configured keyring; thread an optional keyring through `verify`/`why`. The MUST is satisfied by the warning even before real PKI exists.

### F4 — MAJOR: `tests/agree_check.sh` does not test cross-impl *agreement*

The script feeds one schema-invalid record (a float `ts`) to both verifiers and asserts both exit nonzero. That only proves "both reject one obviously-bad input." It never checks that the two impls compute **identical canonical bytes / WarrantIDs / verdicts** for *valid* records, and it exercises none of the canonicalization surface (unicode, escaping, note length). F1 and F2 sail straight through it. The only real cross-impl check that exists is that each impl independently matches the five hardcoded §8 vectors — none of which contain a control char or a multibyte note.

**Fix:** add a differential harness — generate many valid bodies (including adversarial strings/notes/large ints), run both canonicalizers, assert byte-identical WarrantIDs and identical validity verdicts. This is the single highest-value missing test given the project's design rule.

### Minor

- **F5** — Python does not enforce that a `ski@v1` check blob is JCS-canonical; Go does (`canonicalJSON(m) == data`). SPEC §3.1 says the blob is "JCS-canonical." → divergent acceptance of a non-canonical check blob. Fix: Python assert `canon(doc) == raw_bytes`.
- **F6** — `supersede.subject.hash` MUST-equals-the-superseded-WarrantID (SPEC §7) is implemented only as a soft WARN, and only checks the hash is *some* stored WarrantID, not the specific one. A MUST rendered as a warning.
- **F7** — Go implements neither `why` nor any filing/signing path; it is verify-only. Defensible as a "second implementation for WarrantID/verification agreement," but state it as a deliberate scope limit in the README, or fill in `why`.
- **F8** — `is_unverifiable` diverges for an (already schema-invalid) empty-`because` reject: Python `all(... for [])` → `True`; Go → `false`. Low impact but another silent behavior difference.

## Coverage gaps
- No differential canonicalization test (F4) — highest value.
- No negative signature tests (tampered `sig`, wrong `key`, missing actor-signature).
- No ski@v1 negative case (`expect`-mismatch → `fail`, ATP-exhaustion, unresolved blob).
- No `subject.note` boundary test at 200/201 (would have surfaced F2).
- `supersede` path entirely untested.
- Python↔Go **verdict agreement on ski** is never directly compared: Python uses the external oracle, Go uses its embedded reducer; a Go-reducer bug is only caught by the separate `sigma-conformance` vector step, not by agreement with Python. See the cross-project note below.

## Cross-project note (Warrant × Σ-GLYPH)
`ski@v1` is the strongest thing here — a check re-runnable **without trust**. I verified the seam: SPEC §8.2's TV-10 (`atp=20 → expect=H(S)`) matches Σ-GLYPH's TV-10 exactly. But Warrant ships its **own embedded Go Σ-reducer**, independent of the Σ-GLYPH Python reference — so this is *two* independent Σ implementations that MUST agree, and Warrant already has a Python↔Go split in its own canonicalization (F1). Recommend a standing invariant: **the embedded Σ-reducer MUST pass the same Σ-GLYPH machine vectors** (the CI `sigma-conformance` step is the right hook — keep it pinned and expand it), ideally sharing one vector set.
