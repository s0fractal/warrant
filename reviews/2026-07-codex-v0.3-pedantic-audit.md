## Findings

### P0: Oversized integer `ts` creates a Python/Go verification warning split

`SPEC.md` says body numbers are integers and `ts` is a Unix seconds integer, but it does not specify a numeric range. Python accepts arbitrary-size integers and compares them exactly in prior ordering. Go validates any decimal integer string, canonicalizes it into the same WarrantID bytes, but later calls `json.Number.Int64()` in `getInt` and ignores overflow. For very large timestamps, Go clamps both values and can miss a `ts decreases along prior edge` warning that Python emits.

Probe command:

```console
$ python3 - <<'PY'
# temporary signed store: parent ts = 10**100, child ts = 10**50
# then run both verifiers
...
PY
PY rc 0
WARN 3a065c7a6256  binding unverified (no keyring): key 8a88e3dd7409 claims actor a@x
WARN 3a065c7a6256  ts decreases along prior edge 808599dec154
WARN 808599dec154  binding unverified (no keyring): key 8a88e3dd7409 claims actor a@x

verify: 2 records, 0 errors, 3 warnings

GO rc 0
WARN 3a065c7a6256  binding unverified (no keyring): key 8a88e3dd7409 claims actor a@x
WARN 808599dec154  binding unverified (no keyring): key 8a88e3dd7409 claims actor a@x

verify: 2 records, 0 errors, 2 warnings
```

Why this is P0: the bodies are schema-valid under the current spec text, signatures verify, WarrantIDs agree, but the verification outcome differs.

Concrete fix option A, spec-narrowing:

> In §2, replace `ts | integer | MUST | Unix seconds, UTC` with: `ts | integer | MUST | Unix seconds, UTC, in the inclusive range 0..9223372036854775807`. Add: `A body with a negative or out-of-range integer is schema-invalid. Implementations MUST NOT silently clamp or wrap numeric fields.`

Then update Python and Go validators and add fixtures for `ts=-1`, `ts=9223372036854775807`, and `ts=9223372036854775808`.

Concrete fix option B, implementation-widening:

> Keep arbitrary JSON integers legal, but require all ordering comparisons to use exact arbitrary-precision integer comparison. Go must compare `json.Number.String()` as a signed integer using `math/big.Int`, and validation must reject numbers not parseable as exact integers.

Option A is simpler and more appropriate for "Unix seconds".

### P1: §7 "tunnel" text excludes the settling warrant, while implementations and tests include it

`SPEC.md` §7 says: "The record set is the transitive closure of `prior` edges through stored warrants." Read literally, the settling warrant itself is not in its tunnel, only its ancestors are. But both implementations manually include the settling warrant's cited blobs and fingerprints when evaluating re-litigation novelty. `tests/settlement.py` also depends on that behavior: a no-prior settled record blocks a restatement of its own outcome fingerprint.

If a new implementation follows the literal spec, a candidate can cite the exact evidence/check from the settling warrant and appear novel because the settling warrant's own facts are outside the tunnel. If it follows the shipped code, it is not novel.

Concrete text:

> A settling warrant's tunnel is inclusive. Its record set is the settling warrant plus the transitive closure of its `prior` edges through stored warrants. Its blob set is the union of `under`, `evidence`, `subject.hash`, `check`, and `transcript` hashes cited by every record in that inclusive record set.

Also update "within the settling tunnel" wording to say "inclusive settling tunnel" once, then use "tunnel" thereafter.

### P1: Threshold adoption can be satisfied by unbound actor claims even when key state is configured

`SPEC.md` §9 says a root is adopted by an `accept` warrant whose signatures satisfy the adopting root's current settlement policy, including threshold rules. §5.1 says rotation threshold authorization uses already-bound keys, and says bound/unbound remains a report unless a v0.3 policy explicitly makes bound signatures required for settlement-grade verification. The threshold policy grammar names actors, but the spec never says whether threshold satisfaction counts any valid signature claiming that actor or only signatures from that actor's currently bound keys.

The shipped implementations use unbound claims during root adoption. Probe: trust config pins good keys for `a@x` and `b@x`; adoption of root 2 is signed by attacker keys that merely claim `a@x` and `b@x`. Both verifiers activate root 2, emit only unbound warnings, and do not emit `WARN unadopted root`.

Probe output:

```console
$ python3 - <<'PY'
# temporary two-root store; threshold {min_sigs:2, actors:[a@x,b@x]}
# adoption signed by untrusted keys claiming a@x and b@x
...
PY
PY rc 0
INFO 811b193b8466  signature bound: key 8a88e3dd7409 claims actor a@x
WARN c15ed8af0d07  signature unbound: key ed4928c628d1 claims actor a@x
WARN c15ed8af0d07  signature unbound: key ca93ac170518 claims actor b@x
WARN d9907583cffc  signature unbound: key ed4928c628d1 claims actor c@x

verify: 3 records, 0 errors, 3 warnings

GO rc 0
INFO 811b193b8466  signature bound: key 8a88e3dd7409 claims actor a@x
WARN c15ed8af0d07  signature unbound: key ed4928c628d1 claims actor a@x
WARN c15ed8af0d07  signature unbound: key ca93ac170518 claims actor b@x
WARN d9907583cffc  signature unbound: key ed4928c628d1 claims actor c@x

verify: 3 records, 0 errors, 3 warnings
```

Concrete text:

> For settlement-grade threshold evaluation, a signature counts for an actor only if it is cryptographically valid and, when key state is configured for that actor, made by a key currently bound to that actor at that warrant's DAG position. If no key state is configured for a threshold actor, a verifier MAY treat the actor's signature as unbound and MUST NOT count it toward a v0.3 threshold unless the local trust configuration explicitly permits unbound threshold counting.

If the intended design is warning-only even for thresholds, say that explicitly, but then §9 adoption is weaker than "current settlement policy" sounds.

### P1: `well-signed` is defined but not used to gate settlement-active roots

`SPEC.md` §9 defines a root as well-signed if its filing signature is valid and bound where key state is configured. The next sentence defines settlement-active roots by local trust configuration or adoption, but does not require well-signedness. That leaves implementers to guess whether a locally listed root with a bad signature/schema is settlement-active-but-erroneous or excluded from settlement calculations.

The implementations currently build `active_roots` from trusted root IDs before base validation, and `active_records` excludes invalid threshold policy only, not schema/signature errors. Verification still reports those errors, but settlement calculations can have already treated the invalid root's jurisdiction as active.

Concrete text:

> A root is eligible to become settlement-active only if it is well-signed and schema-valid under §6. A locally trusted or adopted root that fails base verification MUST still be reported under §6, but MUST be excluded from settlement and foreclosure calculations until the failure is resolved.

Add a settlement differential test with a trusted genesis root whose signature is tampered: both verifiers should report the signature error and `WARN: unadopted root` or a dedicated `ERR: trusted root is not well-signed`, but must not use descendants for settlement.

### P1: Blob-vs-record resolution rule is contradicted by generic §6 verification behavior

`SPEC.md` §7 says: "A blob hash that is also the WarrantID of a stored record is still a blob reference unless it appears in `prior` or a field whose rule explicitly names WarrantIDs." That is the right disambiguation rule.

But both verifiers' generic unresolved-reference check treats any referenced hash as resolved if it exists either as a blob or as a stored record. That means `under`, `evidence`, `check`, and ordinary `subject.hash` references can be considered resolved merely because a record has that WarrantID, even when the field did not explicitly name WarrantIDs.

Concrete text:

> In §6, split reference resolution by field kind: `prior` MUST resolve to stored warrants; `under`, `evidence`, `check`, `transcript`, and ordinary `subject.hash` MUST resolve to blobs; `subject.hash` MAY resolve to a WarrantID only for rules that explicitly name WarrantIDs, currently supersede subjects and root-adoption subjects. A hash present only as a record does not resolve a blob reference.

Then add a negative/differential fixture where `evidence` equals a stored WarrantID but no blob exists at that hash; both verifiers should warn `unresolved blob`.

## P2 Findings

### P2: README ten-minute flow is stale and fails from a fresh clone

The documented flow uses root-level `policy.txt`, `diff.patch`, and `check.sh`:

```bash
POL=$(warrant policy add policy.txt)
P=$(warrant propose --subject diff.patch --under $POL ...)
R=$(warrant reject $P --check check.sh ...)
```

A clean archive of this repo has none of those files at the root:

```console
$ tmp=$(mktemp -d audit-fresh.XXXXXX)
$ git archive --format=tar HEAD | tar -x -C "$tmp"
$ (cd "$tmp" && for f in policy.txt diff.patch check.sh; do test -e "$f" && echo FOUND "$f" || echo MISSING "$f"; done)
MISSING policy.txt
MISSING diff.patch
MISSING check.sh
```

Running the flow reaches `FileNotFoundError: 'policy.txt'`, then cascades into missing `--under` / missing-prior errors.

Concrete replacement:

```bash
cp examples/policy.txt policy.txt
cp examples/check.sh check.sh
printf 'demo diff\n' > diff.patch
chmod +x check.sh
POL=$(warrant policy add policy.txt)
```

Or change the README commands to use `examples/policy.txt` and `examples/check.sh`, and create `diff.patch` explicitly before `warrant propose`.

### P2: Normative warning strings use colon form but CLIs emit level/id/message columns

`SPEC.md` says verifiers MUST report strings such as `WARN: key-state conflict`, `ERR: invalid threshold policy`, `WARN: unadopted root`, and `WARN: genesis.json unverified`. The actual CLIs emit:

```text
WARN <wid>  key-state conflict
ERR  <wid>  invalid threshold policy
```

This is not a behavioral bug, but it makes "exact string" tests awkward and can cause pedantic downstream mismatches.

Concrete text:

> When this document names report strings, the normative message text is the substring after the verifier's severity and record identifier. CLI output MAY include additional structured fields such as severity and abbreviated WarrantID. The required message texts are `key-state conflict`, `invalid threshold policy`, `unadopted root`, `genesis.json unverified`, and `re-litigation cites nothing new`.

### P2: Test suite has good coverage, but the pedantic edges are not pinned

Current positives:

- `tests/differential.py` covers every control byte, multibyte strings, astral characters, JCS ordering, and note 200/201 canonicalization.
- `tests/negative.py` covers signature tamper, actor mismatch, stripped signatures, body tamper, dangling prior/supersede, and ski verdict lies.
- `tests/settlement.py` covers root adoption, genesis hash pinning, invalid threshold policy, re-litigation fingerprints, key rotation, conflict, and stale replay.

Gaps worth pinning:

- Add schema-level note boundary tests, not only `canon`: signed `note="б"*200` verifies schema-clean; signed `note="б"*201` is schema error in both implementations.
- Add exact `ts` boundary tests from the P0 above: negative, max accepted value, max+1 rejected or exactly compared depending on chosen fix.
- Add "record hash does not satisfy blob reference" fixture for `evidence` / `under` / ordinary `subject.hash`.
- Add unbound-threshold-adoption fixture after the spec decides whether those signatures count.
- Add empty/malformed `settle` candidate tests. `settlement_admissibility` currently evaluates candidate bodies without first requiring candidate schema validity.
- Add deterministic conflict report tests if UI/automation will consume conflict warnings; today tests compare sorted warning message sets, not output order.

Concrete CI/test text:

> Add `tests/pedantic_edges.py` for numeric bounds, schema note bounds, blob-vs-record resolution, candidate schema validity, and unbound threshold adoption. Wire it through `tests/agree_check.sh` and CI.

### P2: CI gates the important suites indirectly, but the step list hides that fact

`.github/workflows/ci.yml` runs conformance, selftests, sigma conformance, `.warrants` verification, `tests/agree_check.sh`, and settlement-grade verification of the repo store. Because `agree_check.sh` runs differential, negative, and settlement, the actual gate is stronger than the visible CI step names imply.

Concrete change:

- Rename `Cross-implementation verification agreement` to `Cross-implementation agreement: differential + negative + settlement`.
- Add a direct `python3 tests/differential.py`, `python3 tests/negative.py`, `python3 tests/settlement.py` trio if maintainers prefer each failure to show as its own GitHub Actions step.

### P2: Sigma pin is precise but has no freshness policy

CI pins sigma-glyph at `01a1def27bb9d482de2502026c06d5365834b36c`, which is good for reproducibility. The repo does not state when or how that pin should be reviewed.

Concrete text for `README.md` or a CI comment:

> The sigma-glyph CI pin is intentionally fixed for reproducible Warrant conformance. Maintainers SHOULD refresh it only when Σ-GLYPH Book I conformance vectors intentionally change, and the refresh PR MUST include the old/new sigma commit, `sigma-conformance` output, and Warrant conformance output.

### P2: Go and Python `settle` command surfaces are not symmetric

Python:

```text
warrant [--store STORE] settle <settling_wid> <candidate_body>
```

Go:

```text
warrant-go settle <store> <settling-wid> <candidate-body.json>
```

This is acceptable for a verify-only Go implementation, but it is another place where README examples should avoid implying a shared CLI surface.

Concrete text:

> Python uses global `--store`; Go verify/settle take the store path as a positional argument because Go is verify-only.

## Existing Reviews: Agree / Disagree / New

### Agree

- Agree with the accepted Codex GOV-001 finding that tunnel closure needed exact record/blob rules. v0.3 now has those rules, but this audit finds one remaining inclusive/exclusive wording mismatch for the settling warrant itself.
- Agree with the accepted root distinction: "well-signed" and "settlement-active" are the right terms. This audit says the shipped §9 still needs one more sentence connecting well-signedness to settlement-active eligibility.
- Agree with the accepted both-keys rotation fix: incoming signature is proof-of-possession, not authorization. The unbound-threshold-adoption finding is the same class of bug at the root-adoption threshold boundary.
- Agree with the DeepSeek/Gemini dispositions on `genesis.json`: shipped `trust-config.json` and `.warrants/genesis.json` are mutually consistent, and both verifiers correctly reject unpinned/tampered genesis use in tests.
- Agree with Opus v0.2 F1/F2/F4 resolution: current differential canonicalization is strong and passes 43/43.

### Disagree

- Mild disagreement with any implication that the implementation gate is now fully closed: the oversized-`ts` probe is a concrete verifier-output split in shipped code.
- Mild disagreement with "threshold checks" being fully implementation-gated: root adoption currently counts unbound actor claims for threshold activation, so the implementation does not yet make the policy/key-state boundary explicit enough.

### New Findings Relative To Prior Reviews

- P0 oversized integer `ts` verification divergence.
- P1 inclusive tunnel mismatch between §7 text and shipped code/tests.
- P1 unbound-threshold root adoption behavior.
- P1 unresolved blob-vs-record resolution mismatch against §7's disambiguation rule.
- Fresh-clone README ten-minute flow failure due to missing root-level demo files.

## Bottom Line

No WarrantID divergence appeared in the required protocol or §8 vectors; the shipped examples and `.warrants` store verify cleanly. However, Warrant v0.3 as shipped has one concrete P0 verification-output divergence on oversized integer timestamps. Fix the numeric range/comparison rule first, then tighten the §7/§9 wording and add the pedantic edge fixtures above.
